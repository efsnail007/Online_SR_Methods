# Клиент-серверное приложение для повышения разрешения кадров видеопотока

## О проекте

Этот проект представляет собой учебное full-stack приложение для повышения разрешения кадров видеопотока в реальном времени. Клиентская часть получает изображение с веб-камеры, уменьшает кадр до выбранного рабочего разрешения и отправляет его на сервер. Сервер обрабатывает кадр алгоритмом суперразрешения, возвращает улучшенное изображение, а клиент отображает результат рядом с исходным потоком и показывает метрики обработки.

Проект реализует задачу из документа [`Efremov_P2kh_Zadanie.doc`](./Efremov_P2kh_Zadanie.doc), где требуется:

1. Разработать клиент-серверное приложение для захвата кадров с веб-камеры на клиенте, повышения разрешения на сервере и отображения результата на клиенте в реальном времени.
2. Провести отладку и тестирование приложения.
3. Подготовить руководство пользователя.

По сути, это демонстрационный стенд для сравнения нескольких подходов к увеличению изображения:

- `Real-ESRGAN x4plus` как основная нейросетевая модель.
- `SRCNN RGB` как более простая сверточная модель.
- `Bicubic` как базовый немашинный метод для сравнения.

## Что умеет приложение

- Захватывать видеопоток с веб-камеры через браузер.
- Покадрово отправлять изображения на backend по HTTP.
- Выполнять апскейлинг на сервере на CPU или GPU.
- Выбирать модель обработки прямо в интерфейсе.
- Менять рабочее разрешение входного кадра для баланса между качеством и скоростью.
- Отображать:
  - текущий FPS обработанных кадров;
  - время инференса на сервере;
  - полное время цикла запроса;
  - выходное разрешение;
  - статус камеры и backend.
- Поддерживать два формата API:
  - передача сырых байтов изображения;
  - передача изображения в base64.
- Запускаться локально и в Docker.

## Практический смысл проекта

Проект полезен как учебный пример сразу в нескольких плоскостях:

- как пример клиент-серверной архитектуры для задач компьютерного зрения;
- как пример интеграции web-интерфейса и ML-инференса;
- как пример организации backend-сервиса вокруг нескольких моделей;
- как пример сравнения качества и скорости разных методов апскейлинга;
- как пример адаптации нагрузки под железо за счет изменения входного разрешения и выбора CPU/CUDA режима.

## Общая архитектура

Система состоит из двух основных частей.

### Frontend

Frontend написан на `React 19` и `Vite`. Он отвечает за:

- доступ к веб-камере через `navigator.mediaDevices.getUserMedia`;
- отображение исходного кадра;
- кодирование текущего кадра в `jpeg` или `png`;
- отправку кадра на сервер;
- получение обработанного изображения;
- отображение результата и технических метрик.

### Backend

Backend написан на `FastAPI` и выполняет:

- прием изображения от клиента;
- декодирование байтов в массив изображения;
- выбор модели обработки;
- запуск инференса;
- кодирование результата в выходной формат;
- возврат изображения и метаданных клиенту.

### ML-слой

ML-часть реализована на `PyTorch` и включает:

- реализацию архитектуры `RRDBNet` для `Real-ESRGAN x4plus`;
- реализацию `SRCNN`;
- отдельный runtime для `bicubic`-масштабирования;
- единый реестр моделей и runtime-объектов.

## Логика работы системы

Полный цикл обработки выглядит так:

1. Пользователь открывает web-интерфейс.
2. Frontend проверяет доступность backend через `/health` и `/models`.
3. Пользователь включает камеру.
4. Браузер получает видеопоток.
5. Каждый кадр уменьшается до выбранного рабочего размера, например `96x54`.
6. Кадр кодируется в `jpeg` или `png`.
7. Frontend отправляет его в `POST /api/v1/upscale`.
8. Backend декодирует изображение и преобразует его в `numpy`-массив в формате `BGR`.
9. Сервис выбирает нужный runtime по `model_id`.
10. Runtime выполняет увеличение изображения.
11. Backend кодирует результат обратно в `jpeg` или `png`.
12. Ответ возвращается клиенту вместе с HTTP-заголовками, где содержатся метрики инференса и размер результата.
13. Frontend показывает обработанный кадр и обновляет статистику.
14. Затем отправляется следующий кадр.

Важно, что приложение не использует непрерывный видеострим через WebSocket или WebRTC data channel. Вместо этого оно работает в режиме последовательной отправки отдельных кадров по HTTP. Это упрощает реализацию и делает архитектуру прозрачной для учебного проекта, но накладывает ограничения на максимальный FPS.

## Почему выбрана именно такая схема

Архитектурные решения проекта логичны для учебной практики:

- разделение на frontend и backend позволяет изолировать интерфейс от тяжелых ML-вычислений;
- серверная обработка позволяет использовать CPU или GPU независимо от возможностей клиентского устройства;
- покадровая HTTP-передача проще для отладки, чем потоковый транспорт;
- реестр моделей упрощает добавление новых алгоритмов без изменения API;
- выдача технических метрик через заголовки позволяет не смешивать бинарное изображение и служебные данные;
- возможность уменьшать входной размер кадра позволяет управлять компромиссом между скоростью и качеством.

## Backend: подробное устройство

### `app/main.py`

Главная точка входа backend-приложения.

Что здесь происходит:

- создается экземпляр `FastAPI`;
- настраивается логирование;
- подключается `CORS`;
- регистрируются маршруты API;
- в `lifespan` создается `SuperResolutionService`;
- на старте загружаются модели из списка `BACKEND_STARTUP_MODEL_IDS`;
- при завершении приложения модели выгружаются из памяти.

Здесь же используется `run_in_threadpool`, чтобы тяжелые синхронные операции загрузки и выгрузки моделей не блокировали event loop FastAPI.

### `app/core/config.py`

Центральный модуль конфигурации.

Он решает несколько задач:

- читает переменные окружения;
- приводит значения к нужным типам;
- собирает итоговый объект `Settings`;
- описывает структуру `ModelConfig`;
- умеет загружать каталог моделей из JSON;
- умеет объединять встроенные и внешние конфигурации моделей.

Встроенно определены три модели:

- `realesrgan_x4plus`
- `srcnn_rgb`
- `bicubic`

Если задан `BACKEND_MODEL_CATALOG_PATH`, конфигурация может быть расширена или переопределена внешним JSON-файлом.

### `app/api/routes.py`

Этот файл описывает HTTP API. Маршруты сгруппированы по трем направлениям:

- системные;
- модельные;
- инференс.

В нем также реализовано:

- получение сервиса из `app.state`;
- преобразование результата обработки в JSON-ответ для base64 API;
- сопоставление внутренних ошибок сервиса с HTTP-кодами `400` и `503`.

### `app/services/super_resolution.py`

Это прикладной сервис верхнего уровня, который связывает API и ML-runtime.

Его обязанности:

- загрузка и выгрузка моделей через `ModelRegistry`;
- выдача информации о моделях;
- обработка входного base64;
- обработка сырых байтов изображения;
- проверка лимита размера входных данных;
- декодирование и повторное кодирование изображений;
- выбор выходного формата;
- формирование итогового результата `EncodedImageResult`.

Именно этот сервис определяет единый сценарий обработки для всех моделей.

### `app/services/model_registry.py`

Реестр моделей отвечает за управление набором доступных runtime-объектов.

Он:

- строит словарь конфигураций моделей;
- проверяет наличие `default_model_id`;
- лениво создает runtime только при первом обращении к модели;
- загружает модели, указанные на старте;
- хранит уже созданные runtime в памяти;
- возвращает метаданные по доступным моделям.

Такой подход позволяет не загружать все нейросетевые модели заранее, если они не используются.

### `app/services/image_codec.py`

Модуль выполняет все базовые операции кодирования и декодирования:

- удаление `data:image/...;base64,` префикса;
- преобразование base64 в bytes;
- преобразование bytes в base64;
- декодирование изображения через OpenCV;
- нормализацию формата ответа;
- кодирование результата в `jpeg` или `png`.

### `app/ml/model_runtime.py`

Это основной слой абстракции над моделями.

Здесь описаны:

- базовый класс `BaseModelRuntime`;
- runtime для `Bicubic`;
- runtime для `RealESRGAN`;
- runtime для `SRCNN`;
- обработка выбора устройства `cpu/cuda/auto`;
- фабрика `create_runtime`, создающая runtime по `kind` и `architecture`.

Ключевая идея этого слоя: API и сервис обработки не знают деталей конкретной модели, а работают с единым интерфейсом `upscale(...)`.

## Реализованные модели

### 1. Real-ESRGAN x4plus

Основная модель проекта.

Особенности реализации:

- используется архитектура `RRDBNet`;
- масштаб сети: `x4`;
- внутри сети применяются `Residual Dense Block` и `Residual in Residual Dense Block`;
- загрузка весов поддерживает разные ключи checkpoint: `params_ema`, `params`, `state_dict`, `raw`;
- на `CUDA` могут включаться:
  - `half precision`;
  - `channels_last`;
  - `TF32`;
  - `cudnn.benchmark`.

Практический смысл:

- дает наиболее качественный результат среди имеющихся методов;
- тяжелее остальных по вычислениям;
- лучше всего раскрывается при запуске на GPU.

### 2. SRCNN RGB

Более простая сверточная модель суперразрешения.

Особенности:

- трехслойная сеть:
  - `Conv 9x9 -> ReLU`
  - `Conv 1x1 -> ReLU`
  - `Conv 5x5`
- работает сразу по всем трем цветовым каналам;
- runtime переводит вход из `BGR` в `RGB`, а затем возвращает обратно;
- поддерживается residual-режим, если он был заложен в checkpoint.

Практический смысл:

- проще и легче, чем Real-ESRGAN;
- удобна как промежуточный вариант для сравнения;
- демонстрирует классический подход к задаче суперразрешения.

### 3. Bicubic

Базовый алгоритм без нейросети.

Особенности:

- реализован через `torch.nn.functional.interpolate`;
- выполняет обычное бикубическое масштабирование;
- практически не требует подготовки;
- полезен как baseline.

Практический смысл:

- дает нижнюю планку качества;
- позволяет сравнивать, насколько нейросетевые методы реально улучшают изображение;
- удобен для быстрых тестов и отладки API.

## Работа с цветом

В проекте отдельно учтен важный технический аспект: OpenCV по умолчанию работает в формате `BGR`, а нейросетевые модели обычно ожидают `RGB`.

Поэтому в runtime:

- входной `BGR` кадр переводится в `RGB`;
- значения нормализуются к диапазону `[0, 1]`;
- после инференса результат переводится обратно в `BGR`;
- итог кодируется в обычное изображение.

Эта логика критически важна. Ошибка в порядке каналов привела бы к искажению цветов. В репозитории есть отдельные тесты, проверяющие корректность работы с RGB-каналами.

## Frontend: подробное устройство

Вся основная логика frontend сосредоточена в `src/frontend/src/App.jsx`.

### Что делает frontend

- хранит URL backend в `localStorage`;
- хранит выбранную модель в `localStorage`;
- поддерживает чтение API URL из query-параметров `api` или `apiBaseUrl`;
- проверяет доступность backend при запуске;
- получает список моделей с `/models`;
- запрашивает доступ к камере;
- рисует входной поток в скрытый canvas;
- кодирует кадр в blob;
- отправляет blob в backend;
- показывает исходный и обработанный кадры;
- считает FPS по временным меткам последних обработанных кадров;
- предотвращает параллельные запросы через `requestInFlightRef`;
- освобождает старые `ObjectURL`, чтобы не накапливать утечки памяти в браузере.

### Почему frontend уменьшает кадр перед отправкой

По умолчанию камера запрашивается как `640x360`, но на сервер отправляется намного меньший рабочий кадр, например `96x54`.

Это сделано специально:

- уменьшается объем передаваемых данных;
- ускоряется кодирование кадра;
- снижается время инференса;
- повышается итоговый FPS;
- при суперразрешении сервер все равно возвращает увеличенный вариант.

Для учебного real-time режима это разумный компромисс.

## API backend

По умолчанию backend работает с префиксом:

```text
/api/v1
```

### `GET /api/v1/`

Проверка, что приложение backend запущено.

Пример ответа:

```json
{
  "message": "Real-ESRGAN backend is running."
}
```

### `GET /api/v1/health`

Возвращает общее состояние сервиса и информацию о текущей модели по умолчанию.

Пример ответа:

```json
{
  "status": "ok",
  "app_name": "Real-ESRGAN Backend",
  "model_loaded": true,
  "model_id": "realesrgan_x4plus",
  "model_name": "RealESRGAN_x4plus",
  "model_kind": "torch",
  "device": "cpu",
  "weights_path": "/abs/path/to/RealESRGAN_x4plus.pth"
}
```

### `GET /api/v1/model`

Возвращает расширенную информацию о модели.

Параметры:

- `model_id` опционален. Если не передан, используется модель по умолчанию.

В ответе могут присутствовать:

- тип модели;
- архитектура;
- устройство;
- масштаб сети;
- путь к весам;
- теги;
- служебные параметры.

### `GET /api/v1/models`

Возвращает список всех доступных моделей и `default_model_id`.

Это endpoint, который frontend использует для заполнения списка выбора модели.

### `POST /api/v1/upscale`

Основной endpoint для бинарной передачи изображения.

Что ожидает:

- тело запроса: сырые байты изображения;
- `Content-Type: application/octet-stream`.

Поддерживаемые query-параметры:

- `outscale` от `0` до `8`;
- `model_id`;
- `output_format`: `jpeg`, `jpg`, `png`;
- `jpeg_quality` от `1` до `100`;
- `png_compression` от `0` до `9`.

Что возвращает:

- бинарное изображение;
- `Content-Type: image/jpeg` или `image/png`;
- набор заголовков:
  - `X-Inference-Time-Ms`
  - `X-Model-Id`
  - `X-Model-Name`
  - `X-Model-Kind`
  - `X-Model-Device`
  - `X-Outscale`
  - `X-Output-Width`
  - `X-Output-Height`

Именно этот endpoint используется браузерным клиентом в режиме реального времени.

### `POST /api/v1/upscale/base64`

Альтернативный endpoint для JSON-взаимодействия.

Тело запроса:

```json
{
  "image_base64": "....",
  "outscale": 4,
  "model_id": "bicubic",
  "output_format": "png",
  "jpeg_quality": 90,
  "png_compression": 3
}
```

Ответ:

```json
{
  "image_base64": "....",
  "content_type": "image/png",
  "inference_time_ms": 15.0,
  "input_width": 8,
  "input_height": 8,
  "output_width": 32,
  "output_height": 32,
  "outscale": 4,
  "model_id": "bicubic",
  "model_kind": "bicubic",
  "device": "cpu",
  "model_name": "Bicubic"
}
```

Этот формат удобен для интеграции с другими сервисами, скриптами и системами, где не требуется отдавать бинарный ответ напрямую.

## Обработка ошибок

В backend реализовано явное сопоставление внутренних исключений и HTTP-кодов:

- `ValueError` -> `400 Bad Request`
- `FileNotFoundError` -> `503 Service Unavailable`
- `RuntimeError` -> `503 Service Unavailable`

Типовые ошибки:

- пустое изображение;
- некорректный base64;
- слишком большой размер файла;
- неизвестный `model_id`;
- отсутствие файлов весов;
- попытка использовать `cuda`, когда она недоступна;
- невозможность декодировать или закодировать изображение.

Важно: одновременно поднимать `backend-cpu` и `backend-cuda` в текущей конфигурации нельзя, потому что оба сервиса публикуют один и тот же backend-порт из `.env`.

## Docker-конфигурация

### `Dockerfile.cpu`

- базируется на `python:3.13-slim`;
- устанавливает `Poetry`;
- ставит зависимости с extra `cpu`;
- копирует backend-код;
- запускает `uvicorn`.

### `Dockerfile.cuda`

Почти идентичен CPU-версии, но:

- устанавливает зависимости с extra `cuda`;
- предназначен для запуска в окружении с доступной NVIDIA GPU.

### `docker-compose.yml`

Содержит три сервиса:

- `backend-cuda`
- `backend-cpu`
- `frontend`

Особенности compose-конфигурации:

- backend использует общий шаблон `x-backend-common`;
- исходники backend монтируются внутрь контейнера;
- frontend также монтирует исходники и `node_modules`;
- в dev-режиме frontend запускается через `npm run dev`;
- backend может запускаться с `--reload`, если `BACKEND_RELOAD=true`.

## Ограничения и слабые места

Для учебного проекта это нормально, но в отчете полезно честно указать ограничения:

- используется покадровая HTTP-отправка, а не настоящий low-latency streaming;
- итоговый FPS ограничен суммой времени кодирования, передачи, инференса и декодирования;
- нет аутентификации и авторизации;
- нет ограничения частоты запросов;
- нет очереди задач и балансировки нагрузки;
- нет постоянного хранения результатов;
- frontend ориентирован только на работу с камерой, а не на загрузку локального видеофайла;
- нет отдельного сравнения качества по метрикам `PSNR`, `SSIM` и т.п.;
- автотесты покрывают backend, но не frontend UI;
- веса моделей лежат прямо в репозитории, что удобно локально, но не всегда удобно для промышленного проекта.
